# SecureOps Lite Clean Code Guide

## Purpose

This file defines code style, naming, structure, and readability rules for this repository.

Use it with `README.md`, `ARCHITECTURE.md`, `SECURITY.md`, `Roadmap.md`, and `AGENTS.md`.

This file does not replace the existing error-handling design. Keep the current package-specific sentinel error pattern.

## General Rules

- Prefer clear names over comments.
- Keep functions focused on one responsibility.
- Keep files scoped to the package responsibility shown in `ARCHITECTURE.md`.
- Avoid unrelated refactors while making a targeted change.
- Do not add abstractions until they remove real duplication or match an existing local pattern.
- Do not hide important behavior behind vague helper names.
- Keep comments rare and useful. Explain why, not what.
- Keep formatting generated by the language formatter: `gofmt` for Go and the Angular/TypeScript formatter for frontend files.

## Repository Structure

Follow the current project layout:

```text
backend-Go/
|-- api/
|   |-- config/
|   |-- context/
|   |-- controller/
|   |-- dto/
|   |-- middleware/
|   |-- model/
|   |-- repository/
|   |-- security/
|   |-- service/
|   `-- utils/
`-- main.go

frontend-angular/
`-- src/app/
    |-- models/
    |-- pages/
    `-- services/
```

Backend dependency direction:

```text
controller -> service -> repository -> database
```

Keep package responsibilities separate:

- `controller`: HTTP binding, route parameters, request body parsing, and response writing.
- `dto`: request and response structs.
- `service`: business validation, ownership checks, and use-case orchestration.
- `repository`: GORM/database reads and writes only.
- `model`: persistence/domain structs.
- `middleware`: Gin middleware.
- `security`: JWT and security-specific helpers.
- `config`: environment-backed configuration.
- `utils`: database helpers and database error translation.

## Go Naming

- Package names use lowercase singular words where practical: `controller`, `service`, `repository`, `middleware`.
- Exported Go names use PascalCase: `AssetController`, `NewAssetService`, `JWTManager`.
- Unexported Go names use camelCase: `assetServiceImpl`, `validateAsset`, `parseID`.
- Keep common initialisms consistently capitalized: `ID`, `URL`, `JWT`, `CVE`, `IP`, `JSON`, `DTO`.
- Constructor names should follow `NewTypeName`: `NewAssetController`, `NewJWTManager`.
- Interface names should describe behavior needed by the consuming layer.
- Avoid single-letter names except for short receivers and tight loops.
- Receiver names should be short but meaningful: `c` for controllers, `s` for services, `r` for repositories.

## Go Files

- Name files after the main concept they contain: `asset_controller.go`, `asset_service.go`, `asset_repository.go`.
- Keep tests beside the package they test with `_test.go`.
- Keep package helpers in normal implementation files, not in `errors.go`.
- Do not move DTOs into controllers.
- Do not move GORM models into DTO files.

## Go Functions

- Keep controllers thin. They should parse input, call services, and write responses.
- Keep service methods free of Gin response-writing logic.
- Keep repositories free of HTTP concerns.
- Prefer early returns for validation and error paths.
- Keep validation close to the layer that owns the rule.
- Use `context.Context` through request-owned context where database or request-scoped work is involved.
- Avoid building SQL with string concatenation or `fmt.Sprintf`.
- Use parameterized GORM calls for values.

## Error Files

Keep the existing error style.

Package `errors.go` files should contain only:

```go
type ServiceError struct {
	Message string
}

func (e ServiceError) Error() string {
	return e.Message
}

var (
	ErrInvalidRequestData = &ServiceError{Message: "invalid request data"}
)
```

Rules:

- Keep sentinel errors package-specific.
- Keep only the error struct, `Error()` method, and sentinel vars in `errors.go`.
- Put error mapping and helper functions in normal implementation files.
- Wrap errors with `%w` when adding context.
- Use `errors.Is` or `errors.As` when checking wrapped errors.

## DTOs And Models

- Request DTO names end in `Request`: `AssetRequest`, `LoginRequest`.
- Response DTO names end in `Response`: `AssetResponse`, `ErrorResponse`.
- Mapping functions should make direction clear: `ToDataModel`, `ToAssetResponseDTO`.
- Models should represent persisted domain data.
- Do not bind browser input directly into GORM models.
- Do not expose sensitive model fields in response DTOs.

## Angular And TypeScript Naming

- Classes use PascalCase: `Assets`, `Login`, `Auth`.
- Interfaces use PascalCase and describe data shape: `Asset`.
- Variables, functions, and properties use camelCase.
- File names use kebab-case or existing Angular style consistently inside a folder.
- Services should have service-oriented names. Prefer `AssetService` over `Asset` for injectable API services.
- Environment exports should use the Angular convention `environment`, not a plural name.
- Avoid `any`. Use typed interfaces, `unknown`, or narrow types.
- Keep API URLs and environment values in environment files, not components.

## Angular Files

- Page components stay under `src/app/pages/<page-name>/`.
- Shared data shapes stay under `src/app/models/`.
- API and state services stay under `src/app/services/`.
- Keep templates, component classes, styles, and specs together for each page.
- Keep components focused on UI state and delegation to services.
- Keep backend API calls inside services.

## Tests

- Test files live beside the code they verify.
- Test names should describe behavior: `TestJWTManagerRejectsInvalidTokens`.
- Prefer table tests when checking multiple similar cases.
- Tests should not require a live database unless they are intentionally integration tests.
- Do not weaken production code only to make tests pass.
- Use fake implementations for interfaces when testing package behavior.

## Clean Code Checklist

Before finishing a change:

- Names clearly describe intent.
- The file belongs in the correct package/folder.
- The change follows existing layer boundaries.
- DTOs, models, services, repositories, and controllers are not mixed.
- `errors.go` files keep the existing sentinel-only style.
- SQL values are parameterized.
- Tests are focused and placed beside the package.
- No unrelated formatting or refactoring was introduced.
