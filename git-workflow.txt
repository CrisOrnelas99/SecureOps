Git Workflow For Future Phases

Goal:
Do new work on a feature branch, open a Pull Request, let CodeRabbit review it, then merge back into master.

Important Rule:
Do not keep building new features directly on master if you want PR reviews.

--------------------------------------------------
Full Workflow
--------------------------------------------------

Step 1: Check repo status
Command:
git status

What this does:
- shows changed files
- shows whether your working tree is clean

Step 2: Switch to master
Command:
git checkout master

What this does:
- moves you back to the master branch before starting new work

Step 3: Pull latest master
Command:
git pull

What this does:
- updates your local master with the newest remote changes

Step 4: Create a new feature branch
Example command:
git checkout -b feature/asset-vulnerability-assignment

What this does:
- creates a new branch
- switches you onto that branch

Branch naming examples:
- feature/asset-vulnerability-assignment
- feature/go-risk-service
- fix/jwt-auth-errors
- chore/update-readme

Step 5: Build the feature on that branch
What this means:
- make your code changes
- test them
- keep commits focused on the feature

Step 6: Check what changed
Command:
git status

Optional useful command:
git diff

What this does:
- shows what files changed
- helps you review before committing

Step 7: Stage changes
Command:
git add .

Optional more selective staging:
git add path/to/file

What this does:
- prepares the changed files for commit

Step 8: Commit changes
Example command:
git commit -m "Add asset-vulnerability assignment endpoints"

What this does:
- saves a snapshot of your work in Git history

Commit message examples:
- Add asset-vulnerability assignment endpoints
- Implement vulnerability CRUD validation
- Fix JWT auth entry point handling

Step 9: Push the branch to GitHub
Example command:
git push -u origin feature/asset-vulnerability-assignment

What this does:
- uploads the branch to GitHub
- links your local branch to the remote branch

Step 10: Open a Pull Request
Where:
- GitHub website
- or GitHub Desktop

PR settings:
- base branch: master
- compare branch: your feature branch

What this does:
- creates the review page for your changes
- gives CodeRabbit a PR to review

Step 11: Make sure CodeRabbit has repo access
Where:
- GitHub Settings
- Installed GitHub Apps
- CodeRabbit
- Configure

What to check:
- this repository is included in CodeRabbit access

Step 12: Wait for CodeRabbit review
Where to look:
- PR conversation page
- Files changed
- Checks/status area

You know CodeRabbit is done when:
- a bot comment appears
- inline code review comments appear
- or a CodeRabbit check/status appears

Step 13: If CodeRabbit does not trigger, retrigger it
Command:
git commit --allow-empty -m "Trigger CodeRabbit review"
git push

What this does:
- creates a tiny new PR event
- often makes CodeRabbit run on the existing PR

Step 14: Review feedback
What to do:
- read CodeRabbit comments
- decide which changes to make
- update your branch if needed

Step 14A: Implement CodeRabbit fixes yourself
What this means:
- open the files CodeRabbit commented on
- understand the problem first
- make the code changes locally
- test the change if needed

Useful commands while fixing:
git status
git diff

If you make fixes:
git add .
git commit -m "Address CodeRabbit review feedback"
git push

Step 14B: Wait for CodeRabbit to re-review
What to do:
- refresh the PR page
- wait for CodeRabbit to finish again
- read any new comments

If more fixes are needed:
git add .
git commit -m "Address additional CodeRabbit feedback"
git push

Repeat the review -> fix -> push cycle until the PR looks good.

Step 15: Merge the PR after review is done
Yes:
after CodeRabbit is done and the PR looks good, merge into master.

Where:
- GitHub PR page

What this does:
- moves your reviewed feature branch changes into master

Step 16: Sync local master after merge
Commands:
git checkout master
git pull

What this does:
- updates your local master with the merged PR changes

Step 17: Optional cleanup
Delete local branch:
git branch -d feature/asset-vulnerability-assignment

Delete remote branch:
git push origin --delete feature/asset-vulnerability-assignment

What this does:
- keeps your branch list clean after the work is merged

--------------------------------------------------
Short Version
--------------------------------------------------

git status
git checkout master
git pull
git checkout -b feature/your-feature-name
git add .
git commit -m "Your commit message"
git push -u origin feature/your-feature-name

Then:
- open PR into master
- wait for CodeRabbit
- review comments
- fix code locally
- git add .
- git commit -m "Address CodeRabbit review feedback"
- git push
- if needed:
  git commit --allow-empty -m "Trigger CodeRabbit review"
  git push
- merge PR into master
- git checkout master
- git pull

--------------------------------------------------
General Useful Commands
--------------------------------------------------

Show current branch:
git branch --show-current

Show status:
git status

Show local changes:
git diff

See commit history:
git log --oneline --decorate --graph -10

Push current branch:
git push

Pull latest changes:
git pull

Create and switch to new branch:
git checkout -b feature/your-feature-name

Switch branches:
git checkout master
