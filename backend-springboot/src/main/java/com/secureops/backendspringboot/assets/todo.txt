 Phase 6 is the Asset backend. This is where the project starts becoming a real security operations application
  instead of just an auth demo.

  The goal:

  - let users create, view, update, and delete assets
  - store those assets in PostgreSQL
  - protect the asset routes with JWT
  - validate asset data so bad or unsafe input is rejected early

  What an “asset” means here:
  An asset is any system or technology item you want to track for security risk, such as:

  - server
  - workstation
  - firewall
  - router
  - database
  - cloud service

  Why this matters in cybersecurity:
  Assets are the foundation of security visibility. You cannot manage vulnerabilities or risk well if you do not
  know what systems exist and who owns them.

  What you will build in this phase:

  - Asset entity
  - asset DTOs
  - AssetRepository
  - AssetService
  - AssetController
  - CRUD endpoints:
      - create asset
      - get all assets
      - get one asset
      - update asset
      - delete asset

  How it will be implemented overall:

  1. Database/model layer
     You create an Asset entity that maps to a database table.
     It will likely include fields like:

  - id
  - name
  - type
  - ipAddress
  - operatingSystem
  - owner
  - criticality
  - riskScore
  - riskLevel
  - timestamps

  This is the persisted source of truth.

  2. DTO layer
     You should not expose the raw entity directly everywhere.
     You will create DTOs for requests/responses so you control:

  - what clients are allowed to send
  - what the API returns
  - validation rules

  This is good security and good API design.

  3. Repository layer
     The repository talks to PostgreSQL through Spring Data JPA.
     You will use it for operations like:

  - save asset
  - find all assets
  - find by id
  - delete by id

  4. Service layer
     This is where business logic goes.
     Examples:

  - validate asset existence before update/delete
  - set default values
  - map DTOs to entity
  - later connect asset logic to vulnerabilities and risk scoring

  This keeps controllers thin and clean.

  5. Controller layer
     The controller exposes REST endpoints like:

  - POST /api/assets
  - GET /api/assets
  - GET /api/assets/{id}
  - PUT /api/assets/{id}
  - DELETE /api/assets/{id}

  Because your security config already protects non-public endpoints, these routes should require JWT
  automatically.

  Security goals in this phase:

  - require JWT on all asset routes
  - validate required fields
  - validate IP address format
  - reject malformed input early
  - avoid returning internal errors or stack traces
  - do not trust frontend input blindly
  - keep dangerous fields like riskScore and riskLevel under backend control when possible

  Important design recommendation:
  Do not let the client fully control calculated security fields long-term.
  For now you may store riskScore and riskLevel, but eventually those should be derived from backend logic and
  the Go risk service, not trusted from frontend input.

  What this phase gives you:

  - real protected business endpoints
  - a meaningful backend domain model
  - a foundation for later vulnerability assignment and risk scoring
  - a strong internship talking point around secure CRUD API design

  What success looks like at the end of Phase 6:

  - authenticated user can create an asset
  - authenticated user can list assets
  - authenticated user can fetch one asset
  - authenticated user can update an asset
  - authenticated user can delete an asset
  - invalid asset input is rejected cleanly
  - unauthenticated access is blocked

  If you want to proceed, the next step is the first implementation chunk:

  1. decide the exact asset fields
  2. create the Asset entity
  3. add validation-friendly field choices from the start
